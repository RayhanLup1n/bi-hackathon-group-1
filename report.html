1. Abstrak

RADAR Pangan (Real-time Anomaly Detection and Recommendation for Pangan) adalah sistem kecerdasan buatan multi-lapis yang dirancang untuk memantau, mendeteksi, dan memberikan rekomendasi intervensi terhadap inflasi harga pangan strategis di Indonesia. Sistem ini dikembangkan dalam konteks BI Hackathon dengan memanfaatkan data harga harian dari portal Bank Indonesia (PIHPS), yang mencakup lebih dari 347.550 rekaman sejak tahun 2020 di 10 kota Jawa Barat dan DKI Jakarta untuk 21 komoditas pangan.

Sistem bekerja melalui tiga lapis pemrosesan yang saling terintegrasi: (1) Forecast Engine berbasis LightGBM Quantile Regression yang memprediksi harga 7 dan 14 hari ke depan dengan akurasi MAPE serendah 3,52%; (2) Detection Engine yang mengombinasikan pemeriksaan batas HET (Harga Eceran Tertinggi), Z-score online, algoritma CUSUM untuk deteksi dini drift harga, dan disparity scoring antar-kota; serta (3) Decision Engine berbasis Large Language Model (NVIDIA Llama 3.3 Nemotron Super 49B) yang menggunakan pola ReAct (Reason + Act) untuk menganalisis konteks, memanggil tool pengambilan data, dan menghasilkan rekomendasi intervensi konkret dalam Bahasa Indonesia.

Seluruh pipeline diorkestrasi melalui FastAPI (http://localhost:8001) dan terhubung ke basis data Supabase PostgreSQL di cloud. Sistem mampu menghasilkan analisis lengkap dalam waktu di bawah 10 detik dan memiliki mekanisme fallback berbasis aturan bila layanan LLM tidak tersedia, sehingga menjamin ketersediaan (zero downtime).

2. Latar Belakang & Permasalahan

Inflasi harga pangan merupakan salah satu tantangan makroekonomi yang paling langsung dirasakan masyarakat Indonesia. Komoditas seperti cabai, bawang merah, dan beras memiliki volatilitas harga yang tinggi — dipengaruhi oleh faktor musiman, cuaca, distribusi logistik, dan perilaku spekulatif pasar. Bank Indonesia melalui portal PIHPS (Pusat Informasi Harga Pangan Strategis) menyediakan data harga harian dari ratusan kota, namun potensi data ini belum dimanfaatkan secara optimal untuk deteksi dini dan pengambilan keputusan berbasis bukti.

Permasalahan yang diidentifikasi meliputi: (a) tidak ada sistem otomatis yang mendeteksi kenaikan harga sebelum melampaui HET; (b) rekomendasi intervensi bersifat reaktif dan tidak mempertimbangkan distribusi harga antar-wilayah; (c) analisis konteks musiman (Ramadan, panen, hari besar) dilakukan secara manual. RADAR Pangan dirancang untuk mengatasi ketiga permasalahan tersebut secara simultan.

❌ Masalah 1

Tidak ada deteksi dini sebelum harga melampaui HET. Petugas baru bereaksi setelah lonjakan terjadi.

❌ Masalah 2

Rekomendasi intervensi tidak mempertimbangkan surplus/defisit antar-kota di pulau yang sama.

❌ Masalah 3

Konteks musiman (Ramadan, hari raya, musim panen) tidak dianalisis secara otomatis.

4. Pipeline Data (ETL)

Sebelum analisis dapat dilakukan, data mentah dari API PIHPS Bank Indonesia harus diekstrak, dibersihkan, dan dimuat ke basis data cloud. Pipeline ETL dikelola oleh Apache Airflow dengan dua DAG terpisah.

4.1 Ekstraksi — BI PIHPS API

Komponen PihpsExtractor melakukan autentikasi menggunakan XSRF Token + Session Cookie (mekanisme antiforgery ASP.NET Core). API mengembalikan data dalam format pivot table (tanggal sebagai JSON key), sehingga parser khusus diperlukan untuk unpivot ke format baris. Rate limiting 2 detik per request diterapkan untuk mencegah throttling.

DAG 1: data_ready_modelling

Muatan historis penuh (2020–2026) untuk training ML. Berjalan satu kali + incremental via checkpoint di raw.pipeline_log. Output: marts.mart_modelling_harga_pangan.

Manual Trigger

DAG 2: data_ready_dashboard

Update harian (D-1) untuk monitoring dashboard. Termasuk health check ke API PIHPS sebelum ekstraksi (fail-fast jika API down). Output: marts.mart_dashboard_harga_pangan.

Daily 07:00 WIB

4.2 Transformasi — dbt

Transformasi data dilakukan oleh dbt dalam dua tahap: Staging (VIEW, 0 MB storage) untuk pembersihan dan normalisasi, dan Mart (TABLE) untuk fitur engineering siap pakai. Model staging menggunakan ROW_NUMBER() untuk deduplikasi berdasarkan natural key (tanggal, comcat_id, kota_id, pasar_nama).

5. Lapis 1 — Forecast Engine (LightGBM Quantile)

Lapis pertama bertanggung jawab memprediksi harga pangan ke depan menggunakan LightGBM Quantile Regression. Berbeda dari regresi biasa yang hanya memprediksi nilai tengah, quantile regression menghasilkan interval prediksi yang memungkinkan estimasi worst-case secara probabilistik.

Mengapa satu model global (bukan per-komoditas)?
Dengan 21 komoditas × 10 kota = 210 deret waktu, pendekatan per-series tidak efisien. Model global tunggal memanfaatkan transfer learning implisit: pola harga cabai di Bandung membantu prediksi cabai di Bekasi. Komoditas dan kota di-encode sebagai fitur integer kategoris.

5.1 Feature Engineering

Fitur dibangun dari dua sumber: (a) mart Supabase yang sudah berisi lag dan rolling statistics dari dbt, dan (b) fitur HET tambahan yang dihitung saat loading di features.py.

6. Lapis 2 — Detection Engine

Lapis kedua mengombinasikan empat mekanisme deteksi yang saling melengkapi. Output akhir adalah DetectionResult yang berisi final_alert_level hasil eskalasi dari semua sinyal.

6.1 HET Threshold Check (Modul A)

Setiap harga aktual dibandingkan dengan HET yang berlaku. Jika prediksi P90 juga melampaui HET, sistem menghasilkan pre-emptive alert sebelum pelanggaran terjadi secara aktual.

🟢 GREEN
Harga < 80% HET
Kondisi normal, tidak ada tindakan
🟡 YELLOW
80% ≤ harga < 100% HET
Mendekati batas, perlu pemantauan
🔴 RED
Harga ≥ 100% HET
Pelanggaran HET, intervensi segera

6.2 Z-score Online Change Point Detection (Modul B1)

Deteksi perubahan mendadak dengan membandingkan rata-rata 14 hari terakhir (recent window) terhadap rata-rata baseline 60 hari sebelumnya. Jika pergeseran melebihi 2σ baseline, sistem menandai sebagai changepoint (→ red). 

# Simplified logic
sigma_shift = (recent_mean - baseline_mean) / baseline_std
is_changepoint = abs(sigma_shift) >= 2.0  # 2σ threshold

6.3 CUSUM Early Warning (Modul B2)

CUSUM (Cumulative Sum Control Chart) adalah metode kontrol statistik klasik yang mengakumulasi deviasi kecil namun konsisten dari baseline. Keunggulan utamanya: mendeteksi drift harga 2–4 hari lebih awal dibanding Z-score, yang memerlukan lonjakan tunggal besar.

Formula CUSUM

CUSUM⁺[t] = max(0, CUSUM⁺[t-1] + zₜ − k)
CUSUM⁻[t] = max(0, CUSUM⁻[t-1] − zₜ − k)

Alarm jika CUSUM⁺ > h atau CUSUM⁻ > h

k = 0,5σ (allowable slack — standar seri ekonomi)
h = 4σ (decision threshold — ARL ≈ 370 false alarms)

Ilustrasi: CUSUM vs Z-score — Deteksi Drift Harga Cabai
CUSUM mendeteksi drift kumulatif lebih awal (kuning) sebelum harga melampaui HET dan sebelum Z-score aktif (merah). Selisih waktu deteksi: 2–4 hari.

6.4 Disparity Scoring (Modul C)

Disparity score mengukur ketimpangan harga komoditas antar-kota menggunakan rasio harga terhadap rata-rata nasional (harga_ratio_nasional dari mart). Skor mendekati 1,0 mengindikasikan distribusi merata; skor tinggi mengindikasikan ada kota dengan surplus dan ada yang defisit — kandidat koordinasi stok antar-wilayah. 

disparity_score = max_ratio - min_ratio
# max_ratio = kota termahal / rata-rata nasional
# min_ratio = kota termurah / rata-rata nasional

7. Lapis 3 — Decision Engine (LLM ReAct Agent)

Lapis ketiga adalah inti pengambilan keputusan sistem. Sinyal dari Lapis 1 dan 2 disampaikan ke NVIDIA Llama 3.3 Nemotron Super 49B yang berjalan via NVIDIA NIM API. Model ini menggunakan pola ReAct — sebuah pendekatan multi-turn di mana LLM secara iteratif berpikir (Reason) lalu mengambil tindakan (Act) dengan memanggil tool pengambilan data kontekstual, sebelum menghasilkan keputusan final.

7.3 Proximity Filtering — Mencegah Rekomendasi Kota Jauh

Tanpa filtering, LLM berpotensi merekomendasikan kota di pulau yang berbeda (mis. Makassar untuk masalah di Jakarta) yang tidak realistis secara logistik. Sistem menerapkan proximity filter berlapis:

Klasifikasi Proximity

NilaiArtiDiizinkan?
same_islandPulau yang sama✓ Ya
neighboringPulau bertetangga✓ Ya (kondisional)
farBeda pulau jauh✗ Tidak

Peta Tetangga Pulau

  • Jawa ↔ Sumatra, Bali
  • Sumatra ↔ Jawa, Kalimantan
  • Kalimantan ↔ Sumatra, Sulawesi
  • Sulawesi ↔ Kalimantan, Maluku
  • Bali ↔ Jawa, NTB

Tool compare_regional_prices hanya mengembalikan kota dengan proximity ∈ {same_island, neighboring}. System prompt secara eksplisit melarang LLM menyebut kota di luar daftar yang dikembalikan tool, bahkan dalam frasa bersyarat ("jika logistik memungkinkan").

7.4 Rule-based Fallback

Jika LLM API tidak tersedia atau mengalami error, sistem beralih ke rule-based scoring otomatis. Keputusan dibuat berdasarkan kombinasi sinyal: tingkat HET breach, keberadaan changepoint, alarm CUSUM, dan disparity score. Fallback menghasilkan rekomendasi template tanpa menyebut kota spesifik. Tidak ada downtime sistem.

AreaImplementasi

SQL Injection Semua query menggunakan parameterized query via SQLAlchemy (tidak ada string concatenation)

Keterbatasan Sistem (MVP)

Cakupan Data Terbatas

Data hanya tersedia untuk 10 kota Jawa Barat + DKI Jakarta. Banten dan Sulawesi Selatan belum ter-load. Cakupan nasional membutuhkan lebih banyak pengambilan data dari PIHPS.

Proxy Musiman Sederhana

Ramadan diidentifikasi dengan proxy bulan 3–5, bukan kalender Hijriah aktual. Ini menyebabkan variasi kecil tiap tahun tidak terbaca dengan tepat.

Model Global Satu Ukuran

Satu model LightGBM untuk semua komoditas dan kota. Komoditas dengan pola sangat unik (mis. ikan laut vs beras) mungkin kurang optimal.

Latensi LLM

Panggilan ke NVIDIA NIM memerlukan 5–15 detik per analisis karena ukuran model (49B parameter). Batch processing diperlukan untuk volume besar.

12. Kesimpulan

RADAR Pangan adalah bukti konsep yang berhasil mengombinasikan tiga pendekatan komplementer — prediksi statistik, deteksi anomali multi-algoritma, dan penalaran LLM berbasis konteks — dalam satu pipeline yang terintegrasi dan siap produksi. Sistem ini menjawab kebutuhan nyata: mendeteksi risiko inflasi pangan secara dini, mengidentifikasi ketimpangan distribusi harga antar-wilayah, dan menghasilkan rekomendasi intervensi yang spesifik, berbasis data, dan dapat langsung ditindaklanjuti oleh pemangku kebijakan.

Dengan MAPE serendah 3,52%, coverage interval probabilistik 90,9%, dan kemampuan CUSUM untuk mendeteksi drift harga 2–4 hari lebih awal, sistem ini melampaui target KPI yang ditetapkan. Integrasi dengan NVIDIA Llama 3.3 Nemotron Super 49B menghasilkan rekomendasi yang kontekstual, akurat secara geografis (berkat proximity filtering), dan mudah dipahami oleh non-teknis.

Langkah selanjutnya adalah memperluas cakupan data ke level nasional, memperkuat frontend monitoring, dan mengintegrasikan sistem ke dalam ekosistem kebijakan TPID (Tim Pengendalian Inflasi Daerah) Bank Indonesia, sehingga RADAR Pangan dapat berkontribusi nyata dalam menjaga stabilitas harga pangan strategis Indonesia.
